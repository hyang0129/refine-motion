# refine-motion — Constitution

## Thesis

refine-motion is the senior-architect-calling-his-dev-team experience for software. Skilled engineers direct one or more autonomous coding agents by voice, mediated by an AI-assisted issue-refinement step that produces the contract between architect and team. The product is that voice-mediated, refinement-gated direction loop — voice as input, refined issue as the contract, autonomous implementation as the deliverable. The delivery surface — typically a mobile client over Tailscale to a cloud or local devbox — is incidental; swap the client, network, or compute substrate and the project is unchanged.

## Laws

## Law 1 — Implementation is gated by refinement: no agent begins autonomous implementation work until the architect has verified that a refined-issue contract captures their intent

**Why:** The architect's contribution is *intent*; everything else is delegable to agents. Intent capture is non-trivial — complexity clouds the architect's spoken request, and a vague "plz fix" left unrefined produces implementation that diverges from what the architect actually wanted. Refinement is the structured pushback that extracts clear intent from unclear voice: an interview that surfaces architect-owned decisions (which patterns to follow, which edge cases to handle, which constraints to honor) before any code is written. The gate's force is **intent verification** — the architect confirms the refined contract captures their intent, and that confirmation is the only acceptance the gate strictly requires. The failure mode this law exists to prevent is **inferred-spec mode**: an agent produces a spec and starts implementation in the same turn, gambling that its inference matches the architect's intent — which collapses refine-motion into the same "plz fix" loop Aider and Claude Code's default just-in-time mode operate in (research-cited). Recommended current-tech concessions, *not law* (same shape as Law 3's recommended visual review): visual spec review by the architect to catch hallucinations in the contract, and PR review by the architect to catch edge cases the agent missed in implementation. Both are concessions to imperfect agents and disappear with a perfect agent — which would produce a clean spec and clean implementation, with the architect's intent verification alone sufficient to gate. **Perfect-agent test:** a perfect agent still needs verified intent before acting; "the agent figures out intent from context" is Claude Code / Aider's current default, but it conflates navigation (which Anthropic's JIT-retrieval principle correctly addresses — finding the right code to look at) with intent capture (which it doesn't address — different axis).

**Rejected Alternative:** Jump-in-and-iterate without upfront intent capture — the architect speaks (or types) a request, an agent immediately begins implementation, and intent is refined through observed code changes rather than through an explicit pre-implementation contract, per Aider's `/voice` (research-cited: transcription "treated as if typed text," directly acted on as an edit instruction) and Claude Code's default just-in-time-retrieval mode (research-cited: Boris Cherny's "do the simple thing first" + JIT retrieval). Rejected because the architect's intent is non-trivial to capture and a vague spoken request produces implementation that diverges from what the architect actually wanted; the architect-team contract metaphor requires the contract to exist *before* implementation, not to be inferred from implementation observed after the fact. Nuance worth recording: **this rejected alt is a different axis from Anthropic's JIT-retrieval principle** — JIT-retrieval addresses *which code the agent looks at to navigate the repo*, which refine-motion doesn't oppose; this law addresses *whether the architect's intent is verified before any code is written*, which is upstream of any retrieval question.

**Anti-pattern:** Two named patterns. **Fast-path that skips refinement** — any product variant or affordance that routes "simple-enough" requests directly to implementation without producing a refined-issue contract for architect verification (e.g., a "quick fix" mode that interprets short voice requests as direct implementation instructions; an agent heuristic that decides "this issue doesn't need refinement"; a length threshold below which the gate is bypassed). **Inferred-spec mode** — an agent produces a spec and starts implementation in the same turn, treating its own spec inference as the contract rather than waiting for the architect's explicit verification. Both bypass the gate's force (intent verification by the architect) and recreate the "plz fix" loop refine-motion exists to replace. Distinction that defines the line: the anti-pattern is about *whether the architect verifies intent before implementation begins*, not about how long refinement takes or how much spec text gets written. A 30-second refinement that ends with "yes, that's what I meant" is the law working as intended; a 30-minute spec generation that ends without the architect's intent verification is a violation. The gate also applies mid-implementation: any blocker that requires contract modification routes back to the architect for re-verification — an agent adapting scope autonomously, pushing through, or deferring to a judge-agent all violate this law at the modification stage.

**Scope:** project-wide; governs all features and all surfaces (mobile, desktop, devbox).

## Law 2 — The architect's only write path is through agents via the walking-meeting interaction; refine-motion exposes no surface for direct keystroke participation (co-coding) or dictated syntax (voice-as-code)

**Why:** refine-motion is an *optional* part of the architect's workflow — picked up for away-from-IDE intent capture, set down whenever the work needs keystrokes — and that optionality is what lets this law be absolute. Every write to the codebase must flow through an agent: the architect speaks intent, the agent writes the code, mediated by the walking-meeting interaction Law 3 establishes; the architect's IDE is the only sanctioned keystroke surface, and it lives outside this product. The failure mode this law exists to prevent is **architectural lock-in to a co-coding affordance**: shipping any keystroke-level feature here (a "tweak this line" hook on a refinement screen, a desktop edit-alongside mode, a devbox-shell tab, a voice-as-code dictation surface) cements architect-as-keystroke-participant as product DNA; future features optimize around it, the trajectory shifts from walking-meeting toward mobile-IDE, and refine-motion collapses into yet-another-agentic-editor competing with the architect's own IDE. The reinforcing failure is surface duplication: a co-coding affordance forces the architect to maintain two overlapping implementation surfaces (their IDE plus refine-motion), erasing the away-from-keys intent-capture positioning that makes the product distinct. **Perfect-agent test:** a perfect agent never requires keystroke-level direction, so a feature designed to receive keystroke direction is optimizing for the agent's failure rather than walking toward its competence.

**Rejected Alternative:** Pair-programming-style co-coding — the architect speaks or types alongside the agent and edits implementation in the same surface where they direct it, per Aider's `/voice` (transcription "treated as if typed text," directly acted on as an edit instruction), Cursor's foreground agent, and GitHub Copilot's inline edit suggestions. Rejected because the architect's contribution is *intent*; once the architect is editing alongside the agent, the architect has stopped delegating and started doing the implementation work themselves. The Aider/Cursor/Copilot framing collapses direction and implementation into one surface; this product holds them apart and routes implementation through the agent, with the architect's IDE — not refine-motion — as the keystroke fallback when needed.

**Anti-pattern:** Two named patterns. **Co-coding** — the architect's keystrokes go into the codebase alongside the agent's: a desktop "edit alongside" mode, a "tweak this line" hook on a refinement screen, a devbox-shell tab inside the refine-motion client. **Voice-as-code** — the architect dictates syntax and it's typed directly into the codebase, importing Aider's `/voice` "treated as typed text" model into the product. Both bypass the walking-meeting write path: instead of the architect speaking intent to the agent and the agent writing code, the architect's input goes straight to the code. refine-motion is not another IDE — the architect's IDE already exists for that role. Distinction that defines the line: this anti-pattern governs the product's *affordances*, not the architect's behavior outside the product. The architect SSH-ing into the devbox to fix something during a walk is not a violation; refine-motion shipping an in-product shell so they don't have to leave is.

**Scope:** project-wide; governs all features and all surfaces (mobile, desktop, devbox).

## Law 3 — The architect-agent interaction is voice-driven and intent-only; no feature requires the architect to read or edit a visual artifact

**Why:** The architect's contribution is intent, not implementation; the spec, code, and diffs are downstream materializations the agent owns. Required visual review regresses the architect to a code-level reviewer role this product does not serve. Visual review is currently *recommended* in two places where current-tier agents reliably fail — spec hallucinations do not auto-fix in implementation, and agents underbuild tests and miss edge cases — but recommendations are a concession to today's tech, not steady-state design. The failure mode this law exists to prevent is **architectural lock-in to the concession**: a feature optimized for visual review on mobile (e.g., an abridged ADR flow with A/B options) cements the concession as product DNA, blocking the walk-toward-perfect-agent trajectory the law protects. Perfect-agent test: any problem that cannot be communicated by words cannot be solved.

**Rejected Alternative:** Editable-spec-everywhere — the contract artifact is exposed for the architect to read and inline-edit on every surface, including mobile, per GitHub Copilot Workspace's "every stage editable, every downstream stage regenerates" stance. Rejected because the architect's contribution is intent; spec text is the agent's output, not the architect's input. The Copilot Workspace framing collapses intent-capture and spec-editing into one surface; this product holds them apart and assigns the spec to the agent.

**Anti-pattern:** Any feature whose interaction *requires* the architect to look at text on a screen for the conversation to proceed. Concrete example: an abridged ADR flow optimized for mobile with simple A/B options — sounds like clever mobile UX, but the act of optimizing the product surface around visual review is itself the violation. The walking meeting doesn't have a screen. Distinction that defines the line: visual review may be *recommended* (spec review for hallucination-catch, final acceptance review for edge-case-catch) — recommendation is not requirement; a feature that cannot proceed without the visual step is the violation, a feature that proceeds with optional/skippable visual aid is not.

**Scope:** project-wide; governs all features and all surfaces (mobile, desktop, devbox).

## Law 4 — refine-motion is developed only for agentically-tractable codebases; non-tractable repos may run the product but are unsupported, and bugs rooted in non-tractability are explicitly out of scope

**Why:** refine-motion's value prop is the architect speaking intent and agents converting it into implementation that aligns with the architect's mental model of the codebase. The structural failure this law guards against is in the nature of intent itself: without standing architectural commitments — conventions, a constitution, prior decisions visible in code — the solution space for any given intent is *effectively unbounded*, and even a perfect agent must make architectural assumptions to land any concrete change. Take "fix this bug": tests pass, the bug is gone — but is the fix consistent with future development? Did it establish or undermine a pattern? Without standing answers, the agent picks one (its own), and aggregated across many issues the codebase drifts toward whatever the agent finds locally optimal rather than toward the architect's vision. Conventions and CLAUDE.md-style constitutions exist precisely to *pre-answer* these architectural questions; an architect cannot re-litigate every architectural assumption per issue via voice — bandwidth dwarfs the intent capture itself. The product-side failure mode this law exists to prevent is **engineering effort spent on accommodations for non-tractable repos** — a "lite mode," fallbacks that strip refinement on messy repos, auto-detecting tractability — which tacitly accept agent-decided drift as the workflow and pull the product away from its target. (Reinforcing current-tech failure: agents are ~90% reliable on tractable repos today; non-tractable would crater that further and overwhelm the team with bug reports rooted in repo non-tractability rather than product defects.) **Perfect-agent test:** a perfect agent in an unstructured repo is *still* forced to assume architectural direction per issue; refine-motion's workflow assumes those assumptions live in the repo *as standing decisions*. Greenfield and sprawl lack them — different products (v0 / Bolt / Lovable for greenfield; refactor tools for sprawl).

**Rejected Alternative:** Works-on-any-repo — the product engages identically regardless of repo state, with the agent compensating for missing structure through lazy retrieval, repo-mapping, or implicit assumptions, per Cursor ("AI code editor for everyone" — no precondition on repo maturity), Claude Code (lazy grep + just-in-time retrieval, no upfront structure required), and Aider (no precondition; default chat mode). Rejected because the architect-calling-team workflow depends on standing architectural commitments the agent can ground intent against; "works-on-any-repo" products either accept agent-filled architectural assumptions (drift over many issues) or push the architect down to file/function-level direction (voice can't carry). Nuance worth recording: **Aider's repo-map (tree-sitter + PageRank ranking of repo structure) is a complementary tool, not the rejected alt itself** — it improves agent grounding *within an already-tractable repo*. refine-motion and Aider's repo-map are compatible; the rejected stance is the one that says repo state doesn't gate engagement.

**Anti-pattern:** Three named patterns, all on the product-development side rather than the user-runtime side. **Lite mode** — any product variant, mode, or branch in the codebase that activates when a repo is detected as non-tractable, stripping refinement, simplifying prompts, or routing voice intent directly to implementation without the architect-agent contract. **Tractability-aware degradation** — auto-detection of repo state that switches behavior (e.g., "if no CLAUDE.md detected, fall back to lazy grep mode"; "if module structure is shallow, skip the spec phase"). Both bake non-tractability into the product as a first-class case, dilute the core flow, and never produce "great on non-tractable." **Triaging non-tractability bugs as product defects** — accepting bug reports whose root cause is repo non-tractability and routing them to product engineering rather than back to the user as a precondition failure; every accepted bug of this shape pulls engineering toward the wrong target. Distinction that defines the line: this law governs *product-development* decisions and *support* decisions, not user behavior. A user running refine-motion on a non-tractable repo and getting confusing output is not the violation; the product team building features (or fixing bugs) to handle that case better is.

**Scope:** project-wide product-development boundary; governs which user-facing scenarios the product team builds and supports. Does not govern user-runtime behavior — users may run refine-motion against any repo; the law governs whether the team treats resulting failures as in-scope problems to solve.

## Rejected Alternatives

- **Mandatory role-separated multi-agent pipeline (coder ≠ reviewer, always)** — evaluated during law drafting; no perfect-agent invariant found. Moved to CLAUDE.md as a current-tech implementation recommendation, not a constitutional law.
- **Blocker-as-separate-law (D7 — any mid-implementation blocker is a contract-modification event requiring architect re-authorization)** — subsumed into Law 1's anti-pattern; D7's apparent contested alternatives either redefine "blocker" into non-existence or are already prohibited by Law 1, making it a tautological extension rather than an independent law.
- **Continuous-session / workday-length engagement model** — refine-motion is a walking-meeting tool (short-session), not a persistent workday interface; see Corollary 1+2.1 (blockers park; architect re-engages via fresh session).
- **Voice UX implementation (full-duplex barge-in vs push-to-talk vs post-processed monologue)** — not constitutionally decided; delegated to implementation. The constitution governs that voice is the architect's interaction mode (Law 3), not how that input is technically captured.
- **Repo tractability detection and scoring** — current tech cannot reliably measure tractability; building detection machinery produces the tractability-aware degradation Law 4 forbids; see Corollary 2+4.1.

## Corollaries

### Corollary 2+3.1 — Spec correction during sign-off is voice-routed through the agent, never inline

**Derived from:** Law 2 + Law 3

**Tension:** When the agent reads back a refined contract for architect sign-off and the architect catches a hallucinated detail, Law 3's concession permits (but does not require) the architect to review the spec text visually; Law 2 forbids any inline edit affordance. Together they force the question: what is the legitimate correction path?

**Stance:** The correction path is always a spoken intent statement routed through the agent ("change the field name to X") — the product exposes no inline edit shortcut, even for a one-word fix. The architect's visual review may surface the error; the response always goes through the walking-meeting write path.

**Anti-corollary:** A lightweight voice-only atomic correction path that is agent-mediated but bypass-shaped — spoken directly to patch the spec without re-entering the full refinement interview — avoids the "full re-refinement cycle for a single hallucinated token" latency trap Law 3 exists to prevent; cost: the bypass-shaped write path sits adjacent to the co-coding surface Law 2 was designed to foreclose.

---

### Corollary 1+3.1 — The refinement gate closes on spoken confirmation; no feature requires the architect to read the contract text

**Derived from:** Law 1 + Law 3

**Tension:** Law 1 requires the architect to verify the refined contract captures their intent; Law 3 forbids requiring the architect to read a visual artifact for the interaction to proceed. Together they force: how does intent-verification work at the gate?

**Stance:** The gate closes on a spoken "yes, that's what I meant" after the agent reads the contract back aloud. No feature may require the architect to visually inspect the contract text for the confirmation to be valid. Visual spec review remains a recommended concession (per Law 3's existing pattern) but is never required for gate closure.

**Anti-corollary:** Law 1 wins on intent fidelity: spoken readback introduces paraphrase drift and reconstructive recall failure; an architect-invoked escape hatch permitting visual inspection when the spoken confirmation feels unreliable preserves genuine verification at the cost of a constrained, architect-initiated regression on Law 3.

---

### Corollary 1+2.1 — Mid-implementation blockers park the agent; the architect re-engages via a fresh session, not an in-product voice interview

**Derived from:** Law 1 + Law 2

**Tension:** Law 1 requires any contract modification discovered mid-implementation to route back to the architect for re-verification; Law 2 forbids any in-product affordance for direct architect input during implementation (even voice-mediated). Together they force: what does the blocker-response surface look like?

**Stance:** Law 2 wins on surface purity. The agent halts and signals the blocker through an ambient channel outside the product; the architect initiates a fresh refinement session when ready. The product does not provide a blocker-response voice interview or in-product re-engagement surface. refine-motion is a short-session tool (a walking meeting, not a continuous workday interface); Law 1 is satisfied by the fresh session that follows the ambient notification, not by an in-product repair surface.

**Anti-corollary:** Law 1 wins on gate immediacy: the blocker triggers an in-product voice interview that produces an updated contract without requiring the architect to re-initiate a full session; cost: the in-product blocker-response surface becomes a repair affordance functionally equivalent to co-coding (Law 2), training architects to expect mid-stream input channels and anchoring the product to a continuous-workday interaction model.

---

### Corollary 2+3.2 — Setup of refine-motion is fully agent-deployable; credential handoff is the only manual step

**Derived from:** Law 2 + Law 3

**Tension:** Law 3's walking-meeting interaction (voice-first, away-from-desk, no required visual artifacts) and Law 2's "all writes flow through agents" together constrain what bootstrapping the product can legitimately require of the architect.

**Stance:** Setup of refine-motion itself is fully agent-deployable end-to-end. The architect's only manual step is credential handoff (API keys, Tailscale auth keys, cloud tokens) — the class of work agents cannot self-bootstrap because authentication requires an external identity grant. Any setup step requiring the architect to SSH, edit config files, run provisioning commands, or inspect deployment logs is a product defect, not an accepted limitation.

**Anti-corollary:** Law 3's concession (recommended visual review is permitted) and Law 2's acknowledgment that the architect uses their IDE when keystrokes are needed together permit some manual setup steps as current-tech accommodations — not every bootstrap task can be agent-directed without a working agent, creating a chicken-and-egg; cost: the product never commits to fully agent-deployable setup as a hard requirement, accepting manual setup as an ongoing "architect goes to IDE" concession.

---

### Corollary 2+4.1 — The product provides tractability guidance; it does not detect, measure, or enforce tractability state

**Derived from:** Law 2 + Law 4

**Tension:** Law 4 requires agentically-tractable repos as a precondition; Law 2 says all writes flow through agents via the walking meeting. When a user's repo is non-tractable, what does the product provide — and does it attempt to assess tractability programmatically?

**Stance:** refine-motion ships documentation and guidance on how to establish agentically-tractable repos (CLAUDE.md conventions, architectural commitment artifacts, standing-decision practices). The architect performs that work. The product does not programmatically detect, score, or enforce tractability — not because it shouldn't in principle, but because current tech cannot reliably measure tractability, and building detection machinery would produce the tractability-aware degradation Law 4's anti-pattern explicitly forbids. This applies to all surfaces: no tractability score, no CLAUDE.md presence indicator, no convention-coverage gauge.

**Anti-corollary:** An agent-driven repo-scaffolding session (CLAUDE.md generation, convention scaffolding, driven by architect voice intent) is Law 2-compatible (writes flow through agents via walking meeting) and removes the highest friction barrier to first use; cost: auto-setup has the product making architectural decisions for the architect's repo (scaffolded conventions are agent-authored, not architect-authored), which undermines the standing-decisions contract Law 4's tractability requirement relies on — the architect cannot verify what they didn't author.

## Review Heuristic

1. Does this change keep Law 1 (intent gate)? Is there a path where an agent acts on an unverified or inferred contract, or where a mid-implementation blocker bypasses the architect?
2. Does this change keep Law 2 (no direct write)? Does it expose any surface — keystroke, dictated syntax, or in-product repair affordance — that lets the architect write to the codebase without routing through an agent via the walking meeting?
3. Does this change keep Law 3 (voice-only interaction)? Does any step *require* the architect to look at text on a screen for the flow to proceed?
4. Does this change keep Law 4 (tractable repos only)? Does it build support, accommodations, or detection logic for non-tractable repos?

If any answer is "no" or "unclear" — the feature needs redesign, not a carve-out.
